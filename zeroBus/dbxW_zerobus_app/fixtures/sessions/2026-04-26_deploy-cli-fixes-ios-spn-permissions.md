## Session: deploy.sh CLI Fixes, iOS SPN Permissions & Lakebase Project Variables

**Date:** 2026-04-26
**Bundle:** dbxW_zerobus_app + dbxW_zerobus_infra + deploy.sh
**Focus:** End-to-end deploy.sh fixes, iOS Bootstrap SPN declarative permissions, target-specific Lakebase project IDs

---

### Context

After the 2026-04-25 JWT auth session, the infra bundle had been deployed and UC setup job run successfully — both SPNs created, all 9 secret scope keys provisioned. The first full `deploy.sh --target dev` run exposed a chain of CLI compatibility and file-upload issues that blocked the app bundle deployment.

---

### Problems & Root Causes

#### 1. Secrets List JSON Parsing Failure

**Error:** `Could not parse secrets list from scope 'dbxw_zerobus_credentials'`

**Root cause (two issues):**
- `2>&1` on the `list-secrets` command merged stderr warnings into the JSON output, corrupting it
- The Python parser expected `{"secrets": [...]}` (old CLI format), but the current CLI returns a flat array `[{...}]`

**Fix (deploy.sh):**
- Changed `2>&1` → `2>/dev/null` on the `list-secrets` line
- Updated the Python parser to handle both JSON formats: `items = data.get('secrets', data) if isinstance(data, dict) else data`

#### 2. Lakebase Project Status Warning

**Error:** `Lakebase project 'dbxw-zerobus-wearables' not found or not accessible`

**Root cause:** `databricks postgres get-project` requires the full resource path `projects/{project_id}`, but deploy.sh was passing just the bare project ID. The `list-endpoints` call on L360 already had the `projects/` prefix — inconsistency.

**Fix (deploy.sh):**
- Added `projects/` prefix to the `get-project` call: `"projects/${project_id}"`
- Changed `2>&1` → `2>/dev/null` on both Lakebase CLI commands

#### 3. Ghost `.bashrc` Upload Error

**Error:** `Error: open fixtures/.bashrc: no such file or directory`

**Root cause:** The `databricks bundle deploy` CLI (running from the web terminal) tries to upload a `fixtures/.bashrc` file that does not exist in the workspace API, FUSE filesystem, or sync snapshot. The file is a phantom entry visible only to the web terminal's FUSE mount — likely a Linux shell artifact. The workspace UI "Deploy" button (which uses server-side APIs) is unaffected.

**Fix:**
- Created `.databricksignore` with patterns: `.bashrc`, `**/.bashrc`, `fixtures/.bashrc`
- Added `.bashrc` to `.gitignore` as well (CLI reads both)

#### 4. Missing iOS SPN Permissions in App Bundle

**Error:** No error — but the iOS Bootstrap SPN lacked `CAN_USE` on the app resource.

**Root cause:** The `zerobus_ingest.app.yml` permissions blocks didn't include the iOS SPN. The SPN's `application_id` is dynamically generated by the infra UC setup job, so it can't be hardcoded.

**Fix (three files):**
- `databricks.yml` (app): Added `ios_bootstrap_spn_id` variable with `default: ""` (deploy.sh overrides via `--var`)
- `zerobus_ingest.app.yml`: Added `- service_principal_name: ${var.ios_bootstrap_spn_id}` / `level: CAN_USE` to all 4 permissions blocks (base, dev, hls_fde, prod)
- `deploy.sh`: Added `read_ios_spn_id()` function that reads the SPN application_id from the secret scope and passes it as `--var ios_bootstrap_spn_id=<value>` to app bundle deploy

#### 5. Shared Lakebase Project ID Across Targets

**Problem:** dev and hls_fde targets share the same workspace but used the same hardcoded Lakebase project ID `dbxw-zerobus-wearables`. When hls_fde deploys, it would collide with dev.

**Fix (infra bundle, following scAuditor pattern):**
- `databricks.yml` (infra): Added `lakebase_project_id` variable with default `dbxw-zerobus-wearables`; hls_fde overrides to `dbxw-zerobus-wearables-hls-fde`
- `wearables.lakebase.yml`: Changed `project_id: dbxw-zerobus-wearables` → `project_id: ${var.lakebase_project_id}`

### Design Decisions

| Decision | Rationale |
| --- | --- |
| `default: ""` for `ios_bootstrap_spn_id` | Allows standalone `bundle validate` without deploy.sh; empty string is harmless in permissions (no grant created) |
| deploy.sh reads SPN ID from scope, passes via `--var` | Keeps permissions declarative in YAML; survives bundle re-deploys without drift |
| Three `.bashrc` patterns in `.databricksignore` | Belt-and-suspenders — bare name, glob, and explicit path covers all CLI globbing implementations |
| Target-specific Lakebase project IDs | Prevents collisions when dev + hls_fde share the same workspace (same pattern as scAuditor) |
| `2>/dev/null` instead of `2>&1` on CLI JSON commands | Prevents stderr warnings from corrupting JSON that gets piped to Python parsers |
