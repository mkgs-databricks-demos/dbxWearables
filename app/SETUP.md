# AppKit setup ‚Äî merging scaffolded plugin skeletons

The `plugins/` subdirectories here were authored **before** `databricks apps init` runs, because the TypeScript source for Tier-1/Tier-2/Tier-3 plugins is part of this repo's design and shouldn't be regenerated by the AppKit CLI every time. Follow this procedure to merge them with the AppKit-generated scaffold.

## Prerequisites

- Node.js v22+ with `npm`
- Databricks CLI ‚â• v0.295.0, authenticated (`databricks auth login` or a profile in `~/.databrickscfg`)
- Lakebase Postgres project (created one-time per [Lakebase Get Started](https://docs.databricks.com/aws/en/oltp/projects/get-started))
- The shared infra bundle deployed and UC setup job run:

  ```bash
  cd ../zeroBus/dbxW_zerobus_infra
  databricks bundle deploy --target dev
  databricks bundle run wearables_uc_setup --target dev
  databricks secrets put-secret dbxw_zerobus_credentials client_secret --string-value "<admin-generated>"
  ```

## Step 1 ‚Äî Bootstrap the AppKit app

From the **repo root** (one level up from here):

```bash
databricks apps init
```

Follow the prompts. Choose:

- Destination: `app/` (this directory ‚Äî the CLI will prompt before overwriting)
- Plugins to include: **server, lakebase, analytics, caching**
- Lakebase project + branch + database: your dev project
- Deploy after creation: No (we want to edit first)

The CLI will create `app.yaml`, `package.json`, `tsconfig.json`, `server/`, `client/`, and install dependencies. **It will leave the `plugins/` directory alone** because it's empty of AppKit-scaffolded content initially, but may prompt about the existing `README.md` and `SETUP.md` ‚Äî keep them.

## Step 2 ‚Äî Merge the Tier-1 / Tier-2 / Tier-3 plugin skeletons

For each plugin directory under `plugins/` that has a `src/` folder (`zerobus`, `wearable-core`, `garmin`):

```bash
# Generate the AppKit plugin scaffold wrappers
npx @databricks/appkit plugin create \
  --placement in-repo \
  --path app/plugins/zerobus \
  --name zerobus \
  --description "Generic ZeroBus SDK writer" \
  --force

# Repeat for wearable-core and garmin. --force merges with existing files;
# files this repo already authored (index.ts, src/*.ts, README.md) win.
```

If `plugin create` overwrites a file you want to keep, check `git diff` and restore.

For the stub plugins (`fitbit`, `whoop`, `oura`, `withings`, `strava`), each currently has only a `README.md`. Create the AppKit scaffold in place when that provider is actually being implemented:

```bash
npx @databricks/appkit plugin create \
  --placement in-repo \
  --path app/plugins/fitbit \
  --name fitbit \
  --description "Fitbit Web API wearable connector"
```

## Step 3 ‚Äî Wire plugins into `server/server.ts`

AppKit's generated `server/server.ts` starts with:

```ts
import { createApp, server, lakebase } from "@databricks/appkit";

await createApp({
  plugins: [server(), lakebase()],
});
```

Extend it to the full plugin list:

```ts
import { createApp, server, lakebase, analytics, caching } from "@databricks/appkit";
import { zerobus } from "./plugins/zerobus";
import { wearableCore } from "./plugins/wearable-core";
import { garmin } from "./plugins/garmin";
// import { fitbit } from "./plugins/fitbit";   // uncomment when promoted from stub
// import { whoop } from "./plugins/whoop";
// import { oura } from "./plugins/oura";
// import { withings } from "./plugins/withings";
// import { strava } from "./plugins/strava";

await createApp({
  plugins: [
    server(),
    lakebase(),
    analytics(),
    caching(),
    zerobus(),
    wearableCore(),
    garmin(),
  ],
});
```

## Step 4 ‚Äî Configure `app.yaml` resources

AppKit generates an `app.yaml` with `postgres` and `sql-warehouse` resource placeholders. Fill in:

```yaml
resources:
  - name: postgres
    database:
      instance_name: <your-lakebase-project>
      database_name: <your-lakebase-database>
  - name: sql-warehouse
    sql_warehouse:
      id: <warehouse-id-from-zeroBus/dbxW_zerobus_infra>

env:
  - name: LAKEBASE_ENDPOINT
    valueFrom: postgres
  - name: DATABRICKS_SQL_WAREHOUSE_HTTP_PATH
    valueFrom: sql-warehouse
  # ZeroBus credentials live in the infra-bundle-provisioned secret scope.
  - name: ZEROBUS_CLIENT_ID
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: client_id
  - name: ZEROBUS_CLIENT_SECRET
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: client_secret
  - name: ZEROBUS_WORKSPACE_URL
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: workspace_url
  - name: ZEROBUS_ENDPOINT
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: zerobus_endpoint
  - name: WEARABLES_BRONZE_TABLE
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: target_table_name
  # Optional ‚Äî Garmin OAuth (populate before enabling the garmin plugin)
  - name: GARMIN_CLIENT_ID
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: garmin_client_id
  - name: GARMIN_CLIENT_SECRET
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: garmin_client_secret
  - name: GARMIN_WEBHOOK_SECRET
    valueFrom:
      secret:
        scope: dbxw_zerobus_credentials
        key: garmin_webhook_secret
```

## Step 5 ‚Äî Deploy

```bash
cd app
databricks apps deploy
```

On first deploy the `wearable-core` plugin's `setup()` hook runs the migrations in `plugins/wearable-core/migrations/`, creating `app_users`, `wearable_credentials`, `wearable_anchors`, `wearable_sync_runs`, and the `appkit_migrations` tracking table.

## Local development

```bash
cd app
npm run dev
```

See the Lakebase plugin's [Local development](https://databricks.github.io/appkit/docs/plugins/lakebase#local-development) guide for the one-time `databricks_superuser` role grant you need for your identity.
